# Debug UI Design Context

## Target audience

The primary users are the swufe-rag student developers integrating modules A–D. The secondary audience is an instructor or teammate observing retrieval behavior during a demo.

## Core jobs

- Ask a scoped policy question and inspect the grounded answer.
- Verify college/cohort filtering and refusal behavior.
- Compare retrieved chunks, cosine scores, source metadata, and exact quotes.
- Open a complete source chunk when an answer or citation looks suspicious.
- Run deterministic Demo data without model downloads, then switch to production providers later.

## Personality and tone

An industrial/editorial diagnostic workbench: light, calm, precise, and evidence-first. It should feel like an instrument panel for policy QA, not a consumer chatbot. Use warm paper neutrals, deep university blue, restrained status colors, sharp separators, and compact typography.

## Constraints

- Static HTML/CSS/JavaScript with no frontend build step or CDN dependency.
- FastAPI debug server; API paths are isolated under `/api/debug` so the final frontend can move independently.
- Desktop-first three-pane workbench, but all critical controls and evidence remain available on narrow screens.
- Keyboard accessible, visible focus, reduced-motion support, safe text rendering, and no secret/debug prompt leakage by default.
