## 1. 本轮解决了什么

### 1.1 查询链路

当前正式链路为：

```text
用户原问题
  -> LLM 只生成语义理解 JSON
  -> 程序归一化与覆盖校验
  -> 程序选择参数化 SQL / RAG / SQL+RAG / 澄清
  -> 生成结构化证据包
  -> LLM 基于证据自然表达
  -> 程序校验课程、学分、学期、引用和原表泄漏
  -> 最终答案、原文件页码、打开页链接、下载链接
```

LLM 不生成或执行 SQL，也不能增加证据包中不存在的学校事实。浏览器提供 Key 时，问题理解和最终表达默认有 LLM 参与；没有 Key 时保留确定性降级能力，用于开发诊断。

### 2.2 必须正确理解的边界

“所有页面已进入全文/向量库”不等于“每一个表格脚注都已经转成可计算 SQL 规则”。本轮新增的审计报告位于：

- `analysis-output/requirement-note-audit/report.json`
- `analysis-output/requirement-note-audit/report.md`

报告将风险分为：疑似不可能最低值、缺页码证据的最低值、仍未知的选修最低值、全文中存在但尚未绑定到结构化模块的规则块。未绑定不代表未入库；它表示该原文可由 RAG 检索，但还不能安全用于 SQL 自动计算。

当前审计候选包括 42 个“未知选修最低值”、102 个“最低值缺少页码证据”和 647 个“结构化最低值大于当前目录学分”的冲突项。647 项中可能同时包含抽取范围不完整、共享模块错配等问题，属于待复核候选，不能直接全部当作原数据错误。

因此生产策略必须保持：

- SQL 覆盖完整且有页码证据时，才可确认最低值或零结果。
- SQL 无规则或覆盖不足时，转表格/政策 RAG 寻找脚注。
- SQL 与 RAG 均无足够证据时明确说明证据不足，不能把“数据库没抽到”说成“学校没有规定”。
